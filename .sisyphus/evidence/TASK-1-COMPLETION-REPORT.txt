KEY ARCHITECTURE FINDINGS
────────────────────────────────────────────────────────────────────────────

CRITICAL ORDER:
  1. logging.config.dictConfig(LOGGING_CONFIG)
  2. structlog.configure(logger_factory=structlog.stdlib.LoggerFactory(), ...)

Why this order matters:
  • dictConfig must run first to create handlers and loggers
  • structlog.configure() must run after to delegate to LoggerFactory
  • Reversing order breaks the integration

INTEGRATION BRIDGE: ProcessorFormatter
  • Acts as logging.Formatter in dictConfig
  • Processes both structlog and stdlib logging entries
  • Uses foreign_pre_chain for stdlib entries
  • Uses processors for structlog entries

DOMAIN SEPARATION: propagate=False
  • Each domain logger has its own handler(s)
  • Setting propagate=False prevents double-logging to root/system handler
  • Result: worker logs go ONLY to worker.log, not to system.log

OUTPUT FORMATS:
  • ConsoleRenderer: Colored, human-readable (development)
  • JSONRenderer: Machine-readable (production/log aggregation)
